# FleetNexus Infrastructure & Production Architecture

This document serves as a high-level architectural reference for how the FleetNexus .NET application is wired up and configured in production.

## 🏗️ High-Level Architecture Overview

The production environment consists of three primary platforms:
1. **Hostinger**: Domain Registration & DNS Management
2. **Render.com**: Application Hosting (Docker Web Services)
3. **Supabase**: Managed PostgreSQL Database Hosting

---

## 🔗 Component Breakdown & Wiring

### 1. Hostinger (DNS & Custom Domain)
* **Role**: Acts as the entry point for end-users visiting the custom domain (`fleetnexus.io` or `www.fleetnexus.io`).
* **Wiring**: 
  * The DNS settings in Hostinger are configured to point to **Render**.
  * Depending on the exact domain configuration, it either uses an `A Record` pointing to Render's static Anycast IP (for root domains) or a `CNAME` record pointing directly to the Render UI Web Service URL (for subdomains like `www`).
  * Render handles the SSL/TLS certificate generation automatically once the traffic reaches their servers.

### 2. Render (Application Hosting)
The application is broken into two distinct, decoupled **Docker Web Services** running on Render's Linux infrastructure.

#### A. The UI Web Service (Frontend)
* **Role**: Serves the Blazor WebAssembly single-page application (SPA) to the user's browser.
* **Technology**: Docker + Nginx (Alpine Linux)
* **Wiring & Configuration**:
  * During the Docker build, the `.NET 10` SDK compiles the Blazor app into static files (`.html`, `.wasm`, `.js`).
  * The final Docker image only contains **Nginx** and these static files (no .NET runtime is needed in the final container).
  * **Nginx** is configured with `gzip_static on;` to natively serve the highly-compressed `.gz` assets generated by the Blazor publish step, drastically reducing bandwidth.
  * **SPA Routing**: Nginx is configured to route all unknown paths to `/index.html` to allow Blazor to handle client-side routing.
  * **Dependencies**: UI stylesheets (like Bootstrap) are loaded via global CDNs (e.g., jsDelivr) to ensure cross-environment reliability and bypass Git case-sensitivity quirks.
  * **API Communication**: The UI uses an `appsettings.json` file which contains the absolute URL of the Render API Web Service. The user's browser makes standard HTTPS requests directly to the API.

#### B. The API Web Service (Backend)
* **Role**: Handles business logic, security, and database communication.
* **Technology**: Docker + ASP.NET Core 10 Web API
* **Wiring & Configuration**:
  * Runs entirely independent of the UI.
  * Configured to accept CORS requests from the UI domain.
  * Connects to the Supabase PostgreSQL database using an injected Environment Variable: `ConnectionStrings__PostgreSQL`.

### 3. Supabase (Database)
* **Role**: Persistent data storage.
* **Technology**: Managed PostgreSQL
* **Wiring & Configuration**:
  * Supabase exposes the PostgreSQL instance securely over the internet.
  * The Render API connects using the standard ADO.NET connection string format (e.g., `Host=...;Port=5432;Database=...;Username=...;Password=...;`).
  * Node.js "URI" style connection strings (`postgresql://...`) are not used, as the .NET `Npgsql` driver explicitly requires the ADO.NET Key-Value pair format.

---

## 🚦 Traffic Flow Summary
1. User types `fleetnexus.io` into their browser.
2. **Hostinger DNS** routes the request to **Render**.
3. **Render UI Service** (Nginx) responds with the Blazor WebAssembly static files (HTML/CSS/JS/WASM).
4. The user's browser executes the Blazor application.
5. The Blazor application (running in the browser) makes secure HTTP REST calls to the **Render API Service**.
6. The **Render API Service** queries **Supabase** via PostgreSQL protocols, processes the data, and returns JSON to the browser.
